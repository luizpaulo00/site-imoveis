1. Install dependencies:
   ```
   npm install browser-image-compression heic2any @dnd-kit/core @dnd-kit/sortable @dnd-kit/utilities
   ```

2. Create `src/lib/validations/image.ts`:
   - Export `MAX_IMAGES_PER_PROPERTY = 15`
   - Export `ACCEPTED_IMAGE_TYPES = ['image/jpeg', 'image/png', 'image/webp', 'image/heic', 'image/heif']`
   - Export `MAX_FILE_SIZE_MB = 15` (raw input limit before compression)
   - Export Zod schema `imageFileSchema` that validates file type and size

3. Create `src/actions/images.ts` following the pattern in `src/actions/properties.ts` (createClient, revalidatePath):
   - `uploadImage(propertyId: string, formData: FormData)`: receives compressed file via FormData, generates UUID filename (`{propertyId}/{uuid}.jpg`), uploads to `property-images` bucket, inserts row in `property_images` table with next position number. If no other images exist for this property, set `is_cover = true` automatically (first photo is cover). Returns the new image record.
   - `deleteImage(imageId: string)`: fetches image record to get storage_path, deletes from Storage, deletes DB row. If deleted image was cover, auto-promote the image with lowest position to cover. Revalidates path.
   - `reorderImages(propertyId: string, imageIds: string[])`: receives ordered array of image IDs, updates each row's `position` to its array index (0-based). Uses simple integer positions (max 15 photos, batch update is trivial).
   - `setCoverImage(propertyId: string, imageId: string)`: within a single query flow, set all images for this property to `is_cover = false`, then set the target image to `is_cover = true`. Revalidates path.
   - All actions use `createClient()` from `@/lib/supabase/server` and include proper error handling returning `{ error?: string }`.
  </action>

<task type="auto">
  <name>Task 2: Create useImageUpload hook with compression pipeline</name>
  <files>src/components/admin/image-manager/use-image-upload.ts</files>
  <action>
Create `src/components/admin/image-manager/use-image-upload.ts` -- a React hook that orchestrates the full client-side image processing and upload flow.

**Hook signature:** `useImageUpload(propertyId: string)` returns:
- `uploadFiles(files: File[])`: main function to process and upload multiple files
- `uploadQueue`: array of `{ id: string, file: File, preview: string, status: 'compressing' | 'uploading' | 'done' | 'error', progress: number, error?: string }`
- `isUploading`: boolean

**Implementation details:**

1. **File acceptance check:** Validate each file against `ACCEPTED_IMAGE_TYPES` and `MAX_FILE_SIZE_MB`. Skip invalid files with toast error (sonner).

2. **HEIC detection and conversion:** Check `file.type` includes 'heic'/'heif' OR `file.name` ends with `.heic`/`.heif`. If HEIC, dynamically import `heic2any` (`const heic2any = (await import('heic2any')).default`) to avoid bundling the 200KB+ decoder for non-HEIC uploads. Convert to JPEG blob, create new File.

3. **Compression:** Use `browser-image-compression` with options:
   - `maxSizeMB: 0.4`
   - `maxWidthOrHeight: 800`
   - `initialQuality: 0.8`
   - `useWebWorker: true`
   - `fileType: 'image/jpeg'`
   - `onProgress: (p) => updateProgress(id, p)` -- maps to the progress bar (0-100%)

4. **Preview generation:** Use `URL.createObjectURL(file)` for instant preview before compression starts. Revoke URLs on cleanup (useEffect return).

5. **Upload:** Create FormData with compressed file, call `uploadImage` server action. During upload step, set status to 'uploading' with indeterminate progress (files are ~400KB, upload is fast).

6. **Concurrency control:** Process files sequentially (one at a time) to avoid memory issues with large batches per Pitfall 5 from research. Use a queue pattern.

7. **State management:** Use functional `setState(prev => ...)` updates to avoid race conditions per Pitfall 2.

8. **Limit enforcement:** Before starting, check current image count + new files against `MAX_IMAGES_PER_PROPERTY`. If would exceed, toast warning and truncate to remaining slots. Show counter "X/15 fotos" info.
  </action>
